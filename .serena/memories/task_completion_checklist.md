# タスク完了時のチェックリスト

## コード品質

### 1. フォーマットとリント
```bash
# 必須: コードフォーマット
pnpm format

# 必須: リントチェック
pnpm lint
```

### 2. TypeScriptエラー確認
- IDE上でTypeScriptエラーがないことを確認
- `tsc --noEmit` でビルドエラーチェック（必要に応じて）

### 3. ビルド確認
```bash
# 本番ビルドが成功することを確認
pnpm build
```

## テスト（現時点では未実装）
- 将来的にテストが実装された際は必ず実行
- テストの追加・更新も検討

## コード品質の確認

### エラーハンドリング
- [ ] エラーケースを適切に処理しているか
- [ ] try-catchで握りつぶしていないか
- [ ] 明確なエラーメッセージを提供しているか

### セキュリティ
- [ ] APIキーや秘密鍵がハードコードされていないか
- [ ] 環境変数から読み込んでいるか
- [ ] 外部入力を検証しているか

### コード品質
- [ ] DRY原則に従っているか（重複コードがないか）
- [ ] 意味のある変数名・関数名を使用しているか
- [ ] コメントで「なぜ」を説明しているか
- [ ] 使用されていないコードを削除したか

## Git操作

### コミット前
```bash
# ステージング確認
git status

# 差分確認
git diff

# コンベンショナルコミット形式でコミット
git commit -m "feat: add new feature"
git commit -m "fix: resolve bug"
git commit -m "refactor: improve code structure"
git commit -m "docs: update documentation"
git commit -m "test: add test cases"
git commit -m "chore: update dependencies"
```

### 注意事項
- [ ] 原子的なコミット（単一の変更に焦点）
- [ ] 明確で説明的なコミットメッセージ（英語）
- [ ] main/masterブランチへの直接コミットは避ける

## ドキュメント
- [ ] 重要な変更はREADME.mdに反映
- [ ] 新機能の使い方を明記
- [ ] 設計判断はコメントまたはドキュメントに記録

## 学習用プロジェクトとしての配慮
- [ ] 他の学習者が理解しやすいコードか
- [ ] 必要に応じてコメントで説明を追加
- [ ] ボーイスカウトルール：元より良い状態にして残す
